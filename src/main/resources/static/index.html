gwanmun — 고정길이 전문 ↔ JSON 왕복

왼쪽에서 필드를 입력해 전문을 만들고(hex 덤프), 오른쪽에 hex를 붙여넣어 JSON으로 되돌립니다. 바이트가 진실입니다.

1) 필드 → 전문 (Build)

DTO 필드를 고정길이 바이트로 인코딩합니다. 숫자=좌측 제로패딩, 문자=우측 공백패딩, 한글=EUC-KR 2byte.

생성된 전문 (hex + 아스키 병기)
아직 생성 전입니다.

2) 전문 → JSON (Parse)

전문 hex를 붙여넣으면 오프셋대로 잘라 EUC-KR로 디코딩하고 패딩을 제거해 JSON으로 돌려줍니다.

파싱된 JSON

4) 게이트웨이 방어 (인증 → 라우팅 → 유량제어)

같은 /api/gateway/balance 앞에 손으로 짠 필터 체인을 세웠습니다. X-API-Key로 인증(없으면 401, 잘못되면 403)하고, 클라이언트별 토큰버킷으로 유량을 제어합니다(용량 초과 시 429 + Retry-After). 아래에서 키를 바꿔 가며 막히는 순간을 봅니다.

5) 가변 전문 왕복 (거래내역 조회 — 길이 헤더 + 레코드 N건)

잔액조회는 고정 61byte였지만, 거래내역 조회 응답은 건수만큼 레코드가 붙어 길이가 매번 다릅니다. 그래서 전문 앞에 4byte 길이 헤더를 붙여 소켓으로 보내고(LengthPrefixedFramer), 받는 쪽은 헤더로 본문 길이를 안 뒤 그만큼 더 모읍니다(2단계). 아래 hex의 노란 4byte가 길이 헤더입니다.

(b) 돌아온 응답 전선 ← 계정계 (길이 헤더 + 가변 본문)

전송 전입니다.

7) 거래 원장 (거래ID 채번 · 3값 상태 · 마스킹)

게이트웨이를 지나간 모든 거래에 거래고유번호(GWMN+U+날짜+일련번호)를 채번하고, 결과를 3값 상태로 원장에 비동기 적재합니다 — SUCCESS(정상 응답) · FAILED(오류 응답·입력 오류) · UNKNOWN(타임아웃 등 응답을 못 받음 — 임의로 실패 처리하지 않습니다). 계좌번호는 저장 직전 마스킹(앞6+뒤4만 노출)됩니다.

지연 계좌 99999999999999는 목업 계정계가 응답을 5초 늦춰 read 타임아웃(3초)을 일으킵니다 — 요청은 계정계에 도달해 처리됐지만 게이트웨이는 응답을 못 받은 상황.
