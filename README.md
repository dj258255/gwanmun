# gwanmun(관문) — 레거시 전문과 현대 REST를 잇는 연계 게이트웨이

> 은행 계정계는 아직도 **고정길이 전문(電文)**과 **TCP 소켓**으로 말하고,
> 모바일·핀테크는 **JSON**과 **HTTP REST**로 말한다. 둘은 직접 대화가 안 된다.
> gwanmun은 그 사이에서 **①통역(전문↔JSON 변환)**과 **②관문(인증·라우팅·유량제어)**을
> 한 흐름으로 처리하는 연계 게이트웨이다. 인젠트 iGate/MCI/FEP + APIM을 한 제품에 합친
> iPaaS 방향의 축소 학습판.

## "이게 API 게이트웨이냐 연계 게이트웨이냐"

둘 다다. gwanmun은 두 층으로 구성된다 — **만들고 싶은 API 게이트웨이를 코어로 두되,
그것만으로는 레거시(전문)를 못 잇기에 연계 통역 층을 아래에 깐다.**

```
[모바일/핀테크]                  [gwanmun]                          [레거시 계정계]
    │ REST/JSON     ┌──────────────────────────────┐  고정길이 전문/TCP  │
    ├──────────────▶│ ① API 게이트웨이 층            │◀───────────────────┤
    │ "잔액 조회"      │   JWT 인증·라우팅·레이트리밋·로깅 │                          │
    │                │ ──────────────────────────── │                          │
    │◀──────────────┤ ② 연계(통역) 층                │───────────────────▶│
    │ JSON 응답        │   JSON↔전문 매핑·TCP↔HTTP 변환   │      전문 응답             │
    │                │   거래코드 라우팅·어댑터          │                          │
    │                └──────────────────────────────┘                          │
```

## 왜 이 조합인가 (정직한 동기)

1. **하고 싶어서** — API 게이트웨이를 직접 만들어 보고 싶었다. 인증·라우팅·유량제어를 프록시
   레벨에서 처리하는 구조를 손으로 짜 보는 것 자체가 목표. (이력서 이전에 학습 동기)
2. **그것만으론 부족해서** — 국내 금융/공공 SI(인젠트 등)의 연계 직무는 API GW보다
   **고정길이 전문 매핑**을 핵심으로 요구한다. API GW만 만들면 "요즘 것끼리 교통정리"에 그치고,
   레거시를 못 잇는다. 통역 층을 더해야 "연계"가 완성된다.
3. **정체성이 하나로 유지돼서** — 두 층 다 "미들웨어(경로 위에서 메시지를 다룬다)"라 한 제품에
   자연스럽게 산다. (DBTower는 경로 밖 관제라 여기 섞으면 안 됨 — 별도 저장소로 분리한 이유)

## 스택

| 층 | 도구 | 이유 |
|---|---|---|
| API GW | Spring Boot(WebFlux 또는 MVC) + JWT | 필터 체인으로 인증·라우팅·유량제어 직접 구현(프레임워크 학습 목적이라 Spring Cloud Gateway 통짜 사용은 지양, 코어는 손으로) |
| 연계 통역 | Java — 전문 스펙(어노테이션/설정) ↔ byte[] ↔ DTO ↔ JSON | 고정길이 파서/빌더가 이 프로젝트의 심장 |
| 통신 | Netty 또는 java.net 소켓(TCP inbound) + WebClient(HTTP outbound) | 프로토콜 변환 실증 |
| 레거시 목업 | 작은 TCP 에코 서버(전문 요청→전문 응답) | 진짜 은행 없이 e2e 재현 |
| 저장/관측 | MySQL(거래 로그) + (선택)DBTower 연동 | JD의 "트랜잭션 모니터링" 대응 |

## 원칙 (DBTower 계승)

1. **정직한 필요** — 층마다 "왜 필요한가" 먼저. API GW는 학습 동기, 통역 층은 시장 요구.
2. **실측 필수** — 전문 hex 덤프까지 포함한 e2e 스크린샷 + docs/VERIFICATION.md.
3. **못 하는 것 표기** — 실제 은행 표준 전문(예: 금융결제원 규격) 전부가 아니라 대표 거래
   몇 종만 구현. "표준 전수 아님"을 정직히.
4. **블로그** — 개선 아크 서사.
