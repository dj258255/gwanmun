# gwanmun(관문) 로드맵 — 상황 주도 개선 아크

> 기능 나열이 아니라 **"어떤 상황에서 무엇이 깨지고, 그래서 무엇을 만드는가"**로 쓴다.
> 각 Phase = (상황) → (한계) → (판단) → (개선) → (라이브 실측·hex 덤프) → (정직한 잔여).

---

## 0. 이 프로젝트가 존재하는 이유 — 한 상황에서 출발

**상황**: 은행에 모바일 잔액조회를 붙여야 한다. 계정계(코어뱅킹)는 **고정길이 전문(電文) + TCP**로만
말하고, 모바일 앱은 **JSON + HTTP REST**로만 말한다. 둘은 직접 대화가 안 되는데, 계정계를 통째로
뜯어고치는 건 현실적으로 감당하기 어려운 규모다.

**판단**: 두 시스템을 못 고치니 **중간에 통역기를 세운다.** 두 층으로 된 미들웨어 —
①**API 게이트웨이**(외부에 열린 통로의 문지기: 인증·라우팅·유량제어) + ②**연계 통역**(전문↔JSON,
TCP↔HTTP 변환). 마이데이터·오픈뱅킹이 정확히 이 구조다(외부 핀테크가 REST로 요청 → 계정계는 전문).

**"관문" = 옛 시스템과 새 앱 사이의 동시통역사 + 검문소.**

### 정체성 경계 (DBTower와 안 섞음)

- gwanmun = **데이터 경로 위에서 메시지를 중계**하는 미들웨어(인라인).
- DBTower = **데이터 경로 밖에서 관찰**하는 관제(아웃오브밴드).
- 성격이 정반대라 별도 저장소. 느슨한 연결(gwanmun 거래 로그를 DBTower가 관측)만 둔다.

### 전제 사실 (기술 배경)

| 전제 | 비고 |
|---|---|
| 국내 금융/공공은 여전히 고정길이 전문 + TCP 연계가 흔하다 | 대외계·코어뱅킹 고정길이 전문 관례 |
| 금융 전문의 한글은 EUC-KR/CP949 인코딩 | 문자 수가 아니라 byte 수로 다뤄야 하는 함정 |
| TCP는 스트림이라 한 전문이 여러 번에 나뉘어 온다 | partial read — 버퍼 누적·경계 관리 필요 |

## 공통 가드레일

1. **전문은 바이트가 진실** — 문자열로 대충 다루지 않는다. byte[] 오프셋·인코딩·패딩·정렬을 명시.
   한글 1자가 EUC-KR 2byte라 "길이 20"은 문자 수가 아니라 바이트 수(함정).
2. **실측** — 모든 Phase는 로컬 e2e(TCP 목업 계정계 포함) + **hex 덤프 스크린샷** + VERIFICATION 절.
3. **정직 범위** — 실제 금융 표준 전문 전수 구현 아님(대표 거래 몇 종). "표준 전수 아님"을 정직히 표기.
4. **읽고 판단 위임** — 관문은 흐름(인증·라우팅·변환)을 통제하되 **비즈니스 로직(잔액 계산 등)은
   계정계에 위임**한다. 새 REST 엔드포인트는 열지만 계산은 안 한다.
5. **커밋 한국어·이모지 금지·블로그 개선 아크.**

## Phase 1 — 전문(電文) 파서/빌더 (완료, 이 프로젝트의 심장)

**상황**: 계정계가 `"0200"+계좌번호[14]+거래코드[4]+...` 형태의 고정길이 바이트를 보낸다.
이걸 `{"account":"...","code":"..."}` JSON으로 바꿔야 앱이 알아듣는다. 반대도 마찬가지.

**한계 인지**:
- **인코딩**: 한글 필드는 EUC-KR 2byte. `String.substring(오프셋, 길이)`로 자르면 **한글이 깨진다** —
  문자 인덱스가 아니라 byte 인덱스로 잘라야 한다.
- **패딩·정렬**: 숫자는 우측 정렬·좌측 제로패딩, 문자는 좌측 정렬·우측 공백패딩이 관례. 혼동 시 파싱 붕괴.

**판단**: 필드 레이아웃을 **스펙으로 선언**(어노테이션: 이름·오프셋·길이·타입·패딩·인코딩)하고,
스펙을 읽어 byte[]↔DTO↔JSON을 양방향 변환하는 엔진을 만든다. 하드코딩 파싱 금지.

**개선**: 전문 스펙 정의(잔액조회 요청 30B·응답 61B) + Parser(byte[]→DTO) + Builder(DTO→byte[]) +
JSON 브릿지(Jackson).

**실측**: 대표 전문 2종 byte[]↔DTO↔JSON **왕복 무손실**(단위 테스트). 한글 EUC-KR 왕복.
잘린 전문·잘못된 길이 → 명확한 예외. hex 덤프 스크린샷.
**잔여**: 파싱만으론 연계 아님(네트워크 없음) — Phase 2.

## Phase 2 — 프레이밍 & 프로토콜 변환 (완료)

**상황**: 파서는 됐다. 이제 TCP로 들어온 전문을 받아 통역하고, 응답을 다시 전문으로 돌려줘야 한다.

**한계 인지**: **TCP는 스트림이라 한 전문이 한 번에 안 온다.** "고정길이 61바이트"인데 일부만 버퍼에
있으면 나머지를 기다려야 한다(partial read). 반대로 두 전문이 붙어 오기도 한다(뭉침).

**판단**: 프레이밍을 명확히 — 필요한 바이트가 다 모일 때까지 버퍼에 누적한 뒤에야 한 전문으로 넘긴다.
순수 `java.net` 소켓으로 직접 구현해 원리를 드러낸다.

**개선**: `FixedLengthFramer`(누적·완성 프레임 추출·뭉침 분리) + `FramedConnection` + 목업 계정계
TCP 서버 + 게이트웨이 왕복(REST→전문 build→TCP→응답 전문 parse→JSON).

**실측**: 두 JVM(앱↔목업계정계) 실제 소켓 왕복. partial read(전문을 나눠 전송) 재조립·뭉침·동시 N건.
요청/응답 전문 hex 덤프 스크린샷.
**잔여**: 외부에 그냥 열면 위험 — Phase 3.

## Phase 3 — API 게이트웨이 필터 체인 + 모듈러 모놀리스 (완료)

**상황**: 이 통로가 외부(핀테크·마이데이터)에 열린다. 아무나 계정계에 요청 못 하게, 과부하 못 주게
문지기가 필요하다.

**판단**: 필터 체인을 **직접 구현**(프레임워크 통짜 대신 손으로, 게이트웨이가 내부에서 뭘 하는지
이해하려고). 순서: 인증 → 라우팅 → 유량제어. 동시에 코드를 **Spring Modulith 모듈**로 경계화한다.

**개선**:
- 인증: API 키 검증 필터(미인증 401 / 잘못된 키 403).
- 라우팅: 경로/메서드 기반 라우트 테이블(모르는 경로 404).
- 유량제어: 클라이언트별 토큰버킷(초과 429 + Retry-After), 단조 시계·버킷 단위 동기화로 동시성 방어.
- 모듈 경계: message·core·gateway·web를 애플리케이션 모듈로 나누고 `ApplicationModules.verify()`로
  순환 없는 단방향 DAG를 코드가 강제하게 한다.

**실측**: 무키 401 / 잘못된 키 403 / 정상 통과 200 / 연속 전송 429 / 모르는 경로 404, 동시성 테스트.
`verify()` 그린 + 모듈 다이어그램.
**잔여**: JWT/OAuth·분산 rate limit 공유는 미구현(단일노드 인메모리) — 정직히 표기.

## Phase 4 — 가변길이 전문 + 커넥션 풀 (완료)

**상황**: 잔액조회는 고정 61byte였지만, 거래내역 조회 같은 응답은 **레코드 건수만큼 길이가 매번 다르다.**
그리고 요청마다 소켓을 새로 여는 건 낭비다.

**한계 인지**: 가변 전문은 본문 길이를 미리 모른다 — 길이 헤더를 먼저 읽어 본문 길이를 확정한 뒤
그만큼 더 모아야 한다(2단계 프레이밍). 커넥션 풀은 동시성·고갈 처리가 까다롭다.

**판단**: 길이 프리픽스 프레이밍(헤더 → 본문 2단계)과 커넥션 풀(최대 크기·유휴 반납·검증·고갈 거절)을
각각 core 모듈에 추가. 비정상 길이는 fail-closed로 방어.

**개선**: `LengthPrefixedFramer`/`LengthPrefixedConnection`, 가변 전문 스펙(고정 헤더 + 레코드 N건),
`ConnectionPool`(스레드 안전·슬롯 선점으로 초과 생성 차단) + 각 클라이언트 레트로핏.

**실측**: 헤더/본문 반쪽 도착 재조립·뭉침·비정상 길이 거절, 가변 레코드 N건 왕복, 풀 재사용(같은 소켓
재사용)·동시성·고갈 정책. `verify()` 계속 그린.
**잔여**: 커넥션 풀은 단일 대상 기준 — 다중 백엔드 풀링은 확장 지점.

## Phase 5 — 관측 가능한 거래 원장 (완료)

**상황**: 통로는 튼튼해졌는데, 게이트웨이가 거래를 흘려보내기만 하고 **아무것도 기억하지 못한다.**
"어제 그 이체 요청이 어떻게 됐나요?"에 답할 방법이 없다 — 앱 로그를 뒤져도 어떤 로그가 어떤 요청인지,
그 요청이 계정계까지 갔는지, 응답을 받았는지 알 수 없다. 장애가 나면 대응 불가다.

**한계 인지**:
- **타임아웃 = 실패가 아니다.** 응답을 못 받은 거래를 임의로 FAILED로 적으면, 실패인 줄 알고 재시도해
  이중 거래가 난다. 계정계에서는 처리됐을 수 있다 — 상태는 2값이 아니라 **3값**이어야 한다.
- **적재가 거래를 막으면 안 된다.** 원장 DB가 죽었다고 거래까지 죽으면 관측이 장애 원인이 된다.
- **원장에 계좌 원문을 쌓으면 안 된다.** 거래 기록이 곧 민감정보 창고가 된다.

**판단**: 새 모듈 `ledger`를 세운다 —
- **거래고유번호 채번**: 오픈뱅킹 bank_tran_id의 자기설명 구조 참조(생성주체 GWMN + 구분 U + 날짜8 +
  일련번호9). AtomicLong(스레드 안전) + 자정 이후 ms 기반 시드(재기동 안전).
- **3값 상태**: 응답 정상 수신=SUCCESS / 명확한 오류 응답·입력 오류·연결 거부=FAILED /
  **타임아웃·응답 없는 끊김=UNKNOWN**. 원인 예외 사슬로 판정.
- **비동기 적재**: 전용 스레드 + 유한 큐. 큐 포화·DB 장애에도 거래는 진행(WARN만).
- **마스킹**: 저장 직전 한 곳에서 앞6+뒤4만 노출. 앱 로그의 계좌도 같은 규칙.
- **correlation ID**: 수신 헤더 승계(없으면 생성) → MDC로 전 로그 라인 + 응답 헤더 + 원장.
- **관측**: actuator 헬스(liveness/readiness 분리) + Prometheus 커스텀 메트릭(토큰버킷 소비/거절,
  풀 active/idle/opened/reused, TCP 왕복 타이머, 거래 상태별 카운터) + graceful shutdown.
- 저장소는 H2(로컬·테스트)/PostgreSQL 컨테이너(운영 데모, docker-compose 포트 25432).

**실측**: 정상→SUCCESS / 없는 계좌(0001)→FAILED / 지연 모드 계좌(목업이 응답을 5초 지연, read
타임아웃 3초)→**UNKNOWN**이 원장에 실제 기록. PostgreSQL 행 직접 확인(계좌는 마스킹 형태만).
/actuator/prometheus 커스텀 메트릭 노출. 화면 스크린샷(3값 색상). `verify()` 계속 그린(5모듈 DAG).

**잔여**: 망취소(취소 전문)·거래 상태 조회(대사) 미구현 — UNKNOWN을 적는 것까지가 이번 범위이고,
UNKNOWN을 **해소**하는 건 다음 단계. 멱등키 없음(같은 요청 재시도 구분 불가). 채번은 단일 노드 전제.

## Phase 6 — 장애 내성 + UNKNOWN 해소 (완료)

**상황**: 두 개의 폭탄이 남아 있었다. 하나, 계정계가 죽거나 느려지면 **모든 호출이 타임아웃까지
스레드를 붙잡아** 게이트웨이가 같이 눕는다(장애 전파). 둘, Phase 5의 원장은 UNKNOWN을 **기록만
하고 방치한다** — "모른다"고 적어 두는 것까지가 전부였고, 모르는 거래를 확정 짓는 절차가 없었다.

**한계 인지**:
- **재시도는 양날이다.** 응답을 못 받은 거래를 무조건 재전송하면, 계정계에서 이미 처리된 변경성
  거래가 두 번 실행된다(이중 거래). 재시도 허용 여부는 옵션이 아니라 거래의 성격(조회성/변경성)에
  못 박아야 한다.
- **소켓 read 타임아웃은 "호출 한 번"의 제한일 뿐이다.** 재시도가 붙으면 거래 하나가 read 타임아웃 ×
  시도 횟수 + 백오프로 늘어진다 — 거래 전체의 데드라인이 따로 필요하고, 마지막 시도의 read 제한도
  남은 시간으로 깎여야 한다.
- **서킷 HALF_OPEN은 동시성 함정이다.** 회복 확인 전에 트래픽을 다 열면 살아나던 백엔드를 다시
  눕힌다 — 탐침 수를 제한하고, 탐침이 나가 있는 동안의 호출은 즉시 거절해야 한다.
- **원거래 전문에 거래고유번호가 없으면 대사가 불가능하다.** 게이트웨이 원장만 아는 ID로는 계정계에
  "그 거래 처리했습니까"를 물을 수 없다 — 열쇠가 전문 본문에 실려야 한다.

**판단**:
- 서킷브레이커를 **직접 구현**(프레임워크 통짜 대신 손으로 — Resilience4j 안 씀). CLOSED→(연속 실패
  임계)→OPEN(즉시 실패 — 원장 FAILED, 계정계 호출 없음)→(대기)→HALF_OPEN(제한 탐침)→CLOSED.
  동시성은 synchronized 한 겹, 상태 전이는 로그 + Prometheus 게이지/카운터로 노출.
- 재시도는 `TransactionKind`(INQUIRY/MUTATION)로 성격을 타입에 박고, **조회성만** 지수 백오프 재시도.
  변경성은 코드가 0회를 강제 — 응답을 못 받으면 UNKNOWN으로 남겨 해소 절차로 넘긴다.
- 거래 단위 데드라인을 소켓 read 타임아웃과 별개로 둔다(기본 5초). 새 시도의 출발선을 막을 뿐
  아니라 매 시도의 read 타임아웃을 남은 시간으로 깎는다.
- UNKNOWN 해소는 금융 연계의 정수를 따른다 — **응답을 못 받은 거래는 "모름"이고, 확정은 조회로
  한다.** 요청 전문 공통 선두를 `전문구분(4)+거래고유번호(22)`로 확장(30→52byte)하고, 목업 계정계에
  인메모리 원장(처리 거래 기억)과 전문 2종을 추가: 거래상태조회(0400 — 처리됨/미처리)와
  망취소(0420 — 원거래 취소, 멱등). 상태조회가 "처리됨"이면 망취소로 무효화해 CANCELED(호출자는
  이미 오류를 받았으므로 성공을 살려두면 양쪽 장부가 어긋난다 — 오픈뱅킹 망취소 관례), "미처리"면
  그제야 FAILED로 확정. 원장에 해소 시각·방법을 남긴다.

**개선**: `CircuitBreaker`/`ResilientExecutor`/`TransactionKind`(core), 해소 전문 4종(message),
`TransactionResolutionService`(gateway), `TransactionStatus.CANCELED`+해소 이력(ledger),
`POST /api/gateway/resolve/{tranId}`+서킷 상태 API+화면 섹션(web). 서킷 열림 응답은 503으로 구분.

**실측**: 계정계 프로세스를 실제로 죽여 연속 호출 → 502(임계 도달, 한 요청의 재시도 3회가 서킷을
엶) → 503×4(즉시 거절, 원장 elapsed_ms=0 FAILED) → 재기동 → HALF_OPEN 탐침 → CLOSED 회복(전이
로그·메트릭). 지연 계좌 UNKNOWN(5.02초 — 데드라인이 마지막 재시도를 깎음) → 상태조회 "처리됨" →
망취소 → 원장 CANCELED. 유실 계좌 UNKNOWN → "미처리" → FAILED. psql로 해소 이력까지 확인.
`./gradlew test` 129건 그린, `verify()` 그린. [VERIFICATION Phase 6절](VERIFICATION.md).
실측 중 함정 둘도 정직 기록: `ddl-auto:update`가 체크 제약을 갱신 안 해 CANCELED UPDATE가 거부된
것(수동 ALTER), 그 실패 순간 이미 나간 망취소를 멱등성이 구해 준 것.

**잔여**: 해소는 수동 API만(주기 대사 배치는 확장 지점). 멱등키 없음. 응답 전문에 거래ID 없음
(동기 소켓 전제). 서킷은 인스턴스 로컬.

## Phase 7 — 감사가 찾은 결함 소탕 + 보안 경화 (완료)

**상황**: Phase 6까지 "기능이 도는 것"을 실측으로 증명해 왔다. Phase 7은 방향을 뒤집어 **전체 코드
감사**를 돌렸다 — 기능이 도는 것과 트래픽 폭주·계정계 이상 응답·자정 재기동 같은 궂은 날에도 옳게
도는 것은 다른 문제다. 감사는 확정 결함 6건(A1~A6)과 보안 구멍 3건(B1·B2·B4)을 냈다.

**한계 인지**:
- **A1 — RuntimeException 하나가 3중 오작동을 만든다.** `PoolExhaustedException`(내부 풀 고갈)이
  RuntimeException으로 모든 처리 경로를 관통했다: ①실행기가 서킷 실패로 계수해 **계정계가 멀쩡한데
  내부 고갈 3연속이면 서킷 OPEN**(오보 — 이후 모든 거래 차단), ②서비스/컨트롤러가 IOException·
  GatewayException만 잡아 **500 + 원장 미기록**(그 거래가 장부에서 증발), ③조회성 재시도도 안 탄다.
  타입이 계약인데, 계약에 없는 타입이 흘렀다.
- **A2 — 왕복 성공 "이후"의 실패도 같은 구멍.** 계정계가 쓰레기 응답을 주면 파싱 예외·
  NumberFormatException이 관통해 500 + 원장 공백. 계정계 이상이야말로 원장에 남아야 하는 사건이다.
- **A4 — isValid()는 로컬 플래그만 본다.** 계정계가 FIN을 보내고 죽은 소켓도 로컬에선 멀쩡해 보인다.
  write는 OS 버퍼로 "성공"한 뒤 read EOF — 변경성(망취소)이 낡은 소켓을 타면 억울한 UNKNOWN.
- **A5 — 채번 시퀀스가 날짜 변경에 리셋되지 않는다.** 자정을 넘겨 살아 있던 프로세스는 "시드 = 자정
  이후 흐른 시간" 불변식을 깬 채 발급하고, 같은 날 시각 시드가 그 구간을 다시 밟으면 충돌 —
  게다가 unique 위반이 비동기 WARN으로 삼켜져 유실이 보이지도 않는다.
- **A6 — getBytes(charset)는 매핑 불가 문자를 조용히 '?'로 바꾼다.** 금융 전문에서 무음 데이터
  손상은 실패보다 나쁘다. NUMERIC 필드의 비숫자도 검증 없이 나갔다.
- **B1/B2/B4 — 기동 로그에 API 키 원문, /api/history는 무인증·무유량제어로 실제 계정계 거래 유발,
  에러 응답에 내부 host:port·예외 원문·입력 원문 에코.**

**판단**:
- **서킷은 백엔드 실패만 센다.** 풀 고갈은 내부 사정 ≠ 계정계 장애 — 계수하지 않고 타입 그대로
  올린다. 재시도도 안 한다(풀이 이미 borrow-timeout을 기다렸고, 과부하 재시도는 부하 증폭).
  HALF_OPEN에서 백엔드까지 못 가 본 허가는 `onAborted()`로 반납한다(탐침 정원 누수 방지).
- **원장 공백 금지 3겹.** 왕복 이후 실패는 전부 클라이언트 예외로 감싸고(원장 FAILED 보장),
  컨트롤러에 최후 방어 catch를 두고, 풀 고갈은 503 + FAILED로 따로 다룬다(주석의 "풀 고갈→FAILED"
  경로가 이제 실제로 도달 가능하다).
- **죽은 유휴 소켓은 수명으로 거른다.** permit 검증 대신 유휴 TTL(반납 후 N초 경과 시 폐기) —
  간단하고 확실하다. 목업 재기동 + 실소켓 테스트로 고정.
- **날짜가 바뀌면 재시드.** 평상시 채번은 락 없는 AtomicLong 그대로, 날짜 전이에만 synchronized.
  원장 유실은 dropped 카운터(gwanmun.ledger.dropped{reason})로 보이는 유실로 만든다.
- **인코딩은 fail-closed.** CharsetEncoder + CodingErrorAction.REPORT — EUC-KR 밖 문자는 '?' 치환
  대신 GwanmunBuildException. NUMERIC 비숫자도 빌드가 거절.
- **실거래 유발 경로는 전부 관문 안으로.** /api/history를 필터 체인에 편입. /api/ledger·
  /api/pool/stats·/api/circuit/stats는 계정계 호출 없는 읽기 전용 관측 경로라 예외로 남긴다 —
  장애 관찰 요청이 유량제어에 막히면 관측이 안 된다(코드 주석에 판단 명시).
- **외부 응답은 일반화 + correlationId.** 내부 host:port·예외 원문은 서버 로그·원장까지만.

**개선**: `ResilientExecutor`(풀 고갈 비계수)·`CircuitBreaker.onAborted()`·`ConnectionPool` 유휴
TTL(+expired 카운터)·`MessageCodec` fail-closed 인코딩/NUMERIC 검증·`TransactionIdGenerator` 재시드·
`TransactionLedger` dropped 카운터·`GatewayService`/컨트롤러 예외 경로 재정리·`ApiKeyRegistry` 로그
경화·`RouteTable`+필터 등록에 /api/history 편입. 목업에 fault 계좌 2종(레코드 비정합 7777…,
쓰레기 건수 필드 6666…)과 지연 설정(mock.delay-ms) 추가.

**실측 (수정 전 → 후, 동일 시나리오 8건 동시 슬로우 계좌)**:
- 수정 전: 고갈 4건 **HTTP 500**(스택 관통) + **원장에 그 4건 없음**(증발) + 서킷 **OPEN**(openedTotal=1)
  → 직후 멀쩡한 계좌가 503 거절(오보). 기동 로그에 API 키 원문. /api/history 무인증 200.
- 수정 후: 고갈 4건 **HTTP 503**("포화 상태" + 거래ID + correlationId) + **원장 9/9건 완결**
  (SUCCESS 4 + FAILED '풀 고갈' 4 + 확인 1) + 서킷 **CLOSED**(openedTotal=0) → 직후 멀쩡한 계좌 200.
  fault 계좌 2종 → 502 + 원장 FAILED(사유 보존). /api/history 무인증 401. 400 응답 입력 에코 제거.
  기동 로그는 클라이언트 id만. `./gradlew test` **147건 그린**(기존 129 + 회귀 18), `verify()` 그린.
  [VERIFICATION Phase 7절](VERIFICATION.md).

**잔여**: A3(서킷 stale 귀속 — OPEN 직전에 나가 있던 호출의 실패/성공이 새 상태에 섞이는 문제,
permit 토큰 방식)은 난이도 대비 실익이 낮아 이번엔 안 했다 — 감사 백로그에 정직하게 남긴다.

## Phase 8 — CI · A3 서킷 stale 귀속 수정 · k6 부하 실측 · Boot 3.5 (완료)

**상황**: 테스트 147건이 로컬에서만 돌고, 부하 앞에서 서킷이 어떻게 되는지 아무도 실측한 적이 없다.
Phase 7은 A3(서킷 stale 결과 귀속)를 "난이도 대비 실익이 낮다"며 백로그로 미뤘다 — 하지만 그건
**부하를 걸어 본 적이 없어서** 실익을 몰랐던 것이기도 하다.

**한계 인지**:
- **A3는 동시성·전이가 겹쳐야 드러난다.** curl 동시 8건 수준으로는 안 보인다. 서킷이 열렸다 닫혔다
  하는 부하 창에서, CLOSED에서 나간 호출이 OPEN→HALF_OPEN 전이 뒤에 늦게 돌아오면 그 결과가 새
  상태에 잘못 귀속된다(성공이 HALF_OPEN을 거짓으로 닫거나, 실패가 탐침 정원을 음수로 깎거나).
- **버그가 수치를 오염시킨다.** A3를 고치기 전에 부하를 재면 서킷 관련 수치가 못 믿을 값이 된다 —
  A3부터 고치고 재야 한다.
- **부하 목표는 "N TPS 달성"이 아니다(우아한형제들식).** 한계 TPS·P95·병목 지점을 드러내는 게 목적.

**판단**:
- **A3 — permit 토큰(세대) 방식.** `acquire()`가 상태 세대를 담은 permit을 발급하고,
  onSuccess/onFailure/onAborted가 permit을 들고 와 세대가 일치할 때만 상태에 반영한다. 상태가 전이할
  때마다 세대가 올라, 이전 세대에서 나간 늦은 결과는 stale로 무시된다. 실행기는 받은 permit을
  `finally`로 정산해 Error 시에도 탐침 정원이 새지 않게 한다.
- **부하 하네스.** 실측용 `loadtest` 프로파일(풀·rate 상향, 목업 지연 0, 재시도 0)을 별도로 두고,
  (a) 정상 부하 상향으로 한계·P95 곡선 (b) 목업에 직접 붙는 벤치마크(`CoreBankingClient` 재사용)로
  게이트웨이 경유 오버헤드 (c) 죽은 백엔드에서 서킷 off/on 대비로 빠른 실패의 값을 잰다. A3 수정 후 측정.
- **CI·LICENSE**는 가장 싸고 큰 격차 — GitHub Actions에서 테스트를 강제하고, 공개 레포에 빠진 MIT 추가.
- **Boot 3.5 업그레이드**는 마지막에, 깨지면 되돌리고 정직히 남긴다는 전제로 시도.

**개선**: `CircuitBreaker.Permit`(세대 토큰)+stale 무시+`staleResultsTotal` 메트릭,
`ResilientExecutor`의 finally 정산, `loadtest` 프로파일, `DirectCoreBenchmark`(core), k6 스크립트
(`loadtest/gw_balance.js`), `.github/workflows/ci.yml`, `LICENSE`, Boot 3.5.4 + Modulith 1.4.3.

**실측**:
- **A3 결정론적 회귀**: 세대 가드를 무력화하면 정확히 A3 두 건만 실패(수정 전/후 대비). 부하 실측
  (c)에서 **staleResultsTotal=197**(15초) — stale 귀속이 이론이 아니라 실제로 벌어졌음이 계수로 증명됐다.
- **한계 TPS·P95**: 무릎 ~10–12k req/s, ~6k까지 p95<1ms, 실패율 전 구간 0%(과부하는 지연으로 degrade).
- **게이트웨이 오버헤드 ≈ 0.21ms/req**. 처리량 병목은 TCP 풀이 아니라 웹 계층(순수 커넥터 ~40k vs 전체
  경로 ~12k req/s — blocking thread-per-request).
- **빠른 실패의 값**: 죽은 백엔드에서 서킷 off는 351 req/s·p50 8.11s로 붕괴, on은 9,425 req/s·p50
  0.68ms로 즉시 거절(27배·약 1.2만 배).
- **Boot 3.5.4 채택**: 파손은 문서 API 한 줄뿐, 적응 후 150건 그린 + verify() 그린 + 앱 기동 정상.
- `./gradlew test` **150건 그린**(기존 147 + A3 회귀 3), `verify()` 그린. [VERIFICATION Phase 8절](VERIFICATION.md).

**잔여**: 부하 수치는 단일 머신 결합 천장(분리된 부하 발생기·전용 서버 절대 성능은 미측정 — 상대 비교가
목적). 멱등키·EOD 대사 배치·DBTower 연계는 백로그에 남는다.

## 감사 백로그 — 다음에 할 것 / 안 하기로 한 것

Phase 7 감사에서 나온 잔여를 한 곳에 모은다. "언젠가"가 아니라 할 것과 안 할 것을 가른다.

### 완료 (Phase 8에서 소진)
- **[x] CI + LICENSE + k6 부하 실측**: 테스트 150건을 GitHub Actions로 돌리고(verify 포함),
  한계 TPS(~10–12k)·P95·게이트웨이 오버헤드(~0.21ms)·빠른 실패의 값을 k6로 수치화. MIT LICENSE 추가.
- **[x] A3 — 서킷 stale 결과 귀속**: acquire()가 세대 permit을 발급하고 onSuccess/onFailure/onAborted가
  토큰을 들고 와, 전이 이전에 나갔던 호출의 늦은 결과를 stale로 무시. 부하에서 staleResultsTotal=197로
  실재 확인. 실행기 finally 정산까지.
- **[x] Spring Boot 3.5+ 업그레이드**: 3.5.4 + Modulith 1.4.3 **채택**(문서 API 한 줄만 적응, 150건 그린).

### 다음에 할 것
- **멱등키(Idempotency-Key)**: 호출자 재전송을 게이트웨이가 구분하는 흐름 — IETF
  draft-ietf-httpapi-idempotency-key-header 참조.
- **EOD 대사 배치**: UNKNOWN을 주기적으로 훑어 상태조회·망취소로 해소하는 스케줄러(지금은 수동 API).
- **DBTower 연계**: 거래 원장·서킷 상태를 관제가 관측하는 느슨한 연결.

### 안 하기로 한 것
- **JWT/OAuth 단독 인증**: 서버 간 게이트웨이 인증은 API 키+mTLS가 관례고, 토큰 발급 인프라가 이
  프로젝트의 학습 목표(전문·연계)와 겹치지 않는다.
- **분산 rate limit(Redis 등)**: 단일 인스턴스 전제를 유지 — 인스턴스 로컬 토큰버킷으로 개념은
  이미 증명했고, 분산화는 저장소 의존만 늘린다.
- **HA·이중화·중앙 채번**: 단일 노드 경계를 코드 주석·문서에 명시하는 쪽을 택했다 — 요구가 생기면
  채번(노드 식별자/DB 시퀀스)부터 확장.
- **MQ 어댑터(Kafka 등)**: 동기 TCP 전문 연계가 주제다 — 비동기 채널은 응답 전문 거래ID 매칭 등
  전제가 달라져 별개 프로젝트 규모가 된다.
- **금융 표준 전문 전수·전문 암호화·전용선**: 표준 규격 전수와 망 보안(VPN·암호화)은 범위 밖 —
  "표준 전수 아님"을 정직하게 표기한다.

## 실무 실태 조사 (2026-07-09 웹 리서치) — 로드맵의 근거

"다음에 뭘 만들까"를 실무 연계 운영자들이 실제로 부딪히는 문제로 정하기 위해 조사했다.

### 이 도메인이 유효하다는 근거

- **가장 가까운 1차 사료**: 당근페이가 은행·카드·VAN 연동용 TCP Gateway를 직접 구축한
  후기 — 겪은 문제가 이 프로젝트의 Phase들과 그대로 겹친다(기관의 동시 세션 수 제한 →
  풀 크기를 계약값에 일치, 기관이 커넥션을 임의로 끊음 → 주기 헬스체크·자동 재수립,
  전문 분할 도착 → 길이 필드 프레이밍, 타임아웃 → 처리결과조회, **대사조정·재처리를
  게이트웨이 부속 기능으로 내장**).
  [확장성 있는 TCP 통신 시스템 구축하기](https://medium.com/daangn/확장성-있는-tcp-통신-시스템-구축하기-981c230cdc77)
- **타임아웃 3값 처리는 도메인 공식 관례**: 은행 공동망 에러코드 42 = "(TIME OUT)
  처리결과조회로 거래성립여부 확인요망", 오픈뱅킹도 무응답 시 결과조회 절차를 규정 —
  Phase 5~6의 UNKNOWN·상태조회 설계와 일치.
  [오픈뱅킹 입금이체](https://developers.kftc.or.kr/dev/openapi/open-banking/deposit)
- **FEP 장애의 실제 병목은 관측성**: 기업은행 대외계 오작동으로 뱅킹 3시간 중단 —
  기사에 명시된 문제가 "복잡한 규칙 구조로 장애 복구에 긴 시간 소요". 어느 전문이
  어디서 죽었는지 즉시 추적하는 능력이 곧 복구 시간이다(Phase 5 correlation ID·원장의
  방향이 맞다는 방증). [기업은행 FEP 재구축 기사](https://m.bikorea.net/news/articleView.html?idxno=36749)
- **전문은 사라지지 않고 번역 계층이 두꺼워지는 중**: ISO 20022 전환기에 다수 은행이
  코어를 못 고쳐 미들웨어로 MX↔레거시 고정길이를 변환하는 전략을 택했고, 한국은행도
  한은금융망에 ISO 20022 적용을 진행(2026 가동 목표) — 변환 게이트웨이의 문제 공간은
  오히려 확대. [Thoughtworks ISO 20022](https://www.thoughtworks.com/en-us/insights/articles/becoming-ISO-native-navigating-end-of-coexistence-in-ISO-20022-migration-journey),
  [한국은행 ISO 20022](https://www.bok.or.kr/portal/main/contents.do?menuNo=201665)

### 실무 근거로 재확정한 우선순위

| 실무 고통 (근거) | 다음에 만들 것 |
|---|---|
| 대사 불일치가 신뢰도를 결정 — 취소·미확인 건에서 잘 남 ([포트원](https://blog.portone.io/opi_reconciliation/), 당근페이도 내장) | **EOD 대사 배치** — 기관 측 거래 내역(목업 생성) vs 원장 전량 대조, 불일치 유형 분류(양쪽있음/우리만/저쪽만/금액상이), 미확인 자동 해소 |
| 중복 방지는 서버가 강제 — 오픈뱅킹 중복 거래고유번호 = A0326 거절, 토스 Idempotency-Key+409 규격 ([토스 멱등키](https://docs.tosspayments.com/reference/using-api/idempotency-key)) | **멱등키/중복 거절** — 중복 수신 시 원응답 재반환 또는 거절코드, 처리중 동시요청 409 |
| 시뮬레이터는 부수물이 아니라 1급 구성요소 — 금결원도 테스트베드를 별도 제공 | **계정계 시뮬레이터 강화** — 지연·무응답·중간 끊김·중복응답 주입(fault 계좌는 Phase 7에서 시작됨) |
| 성능 목표는 업계 표준 수치가 아니라 피크 실측 역산이 관례 ([우아한형제들](https://techblog.woowahan.com/2572/)) | **k6 부하 실측** — 목표는 "N TPS 달성"이 아니라 한계 TPS·P95·병목 지점 규명. "금융권 표준 P95" 같은 수치는 공개 근거가 없으므로 지어내지 않는다 |

### 하지 말 것의 실무 근거 (기존 판단 보강)

- **가상스레드 전면 전환 금지**: 카카오페이 실측 결론이 "기본 off, I/O 무거운 구간만
  상한 걸린 executor로 선별 적용" — 무제한 수용이 커넥션 풀 고갈·pinning을 유발
  ([카카오페이 가상스레드](https://tech.kakaopay.com/post/ro-spring-virtual-thread/)).
  실험+계측+채택 안 한 이유 문서화가 오히려 실무형.
- **HA 절체**: 실무에서 절체는 L4/인프라 계층 장치 — 앱이 담당하는 부분집합
  (graceful shutdown·재접속)은 이미 Phase 5·백로그로 커버.
- **분산 rate limit**: 실무 답도 "로컬 근사 + 임계 근처만 Redis 검증" 하이브리드 —
  단일 인스턴스에선 존재하지 않는 문제.
