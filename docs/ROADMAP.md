# gwanmun(관문) 로드맵 — 상황 주도 개선 아크

> 기능 나열이 아니라 **"어떤 상황에서 무엇이 깨지고, 그래서 무엇을 만드는가"**로 쓴다.
> 각 Phase = (상황) → (한계) → (판단) → (개선) → (라이브 실측·hex 덤프) → (정직한 잔여).

---

## 0. 이 프로젝트가 존재하는 이유 — 한 상황에서 출발

**상황**: 은행에 모바일 잔액조회를 붙여야 한다. 계정계(코어뱅킹)는 **고정길이 전문(電文) + TCP**로만
말하고, 모바일 앱은 **JSON + HTTP REST**로만 말한다. 둘은 직접 대화가 안 되는데, 계정계를 통째로
뜯어고치는 건 현실적으로 감당하기 어려운 규모다.

**판단**: 두 시스템을 못 고치니 **중간에 통역기를 세운다.** 두 층으로 된 미들웨어 —
①**API 게이트웨이**(외부에 열린 통로의 문지기: 인증·라우팅·유량제어) + ②**연계 통역**(전문↔JSON,
TCP↔HTTP 변환). 마이데이터·오픈뱅킹이 정확히 이 구조다(외부 핀테크가 REST로 요청 → 계정계는 전문).

**"관문" = 옛 시스템과 새 앱 사이의 동시통역사 + 검문소.**

### 정체성 경계 (DBTower와 안 섞음)

- gwanmun = **데이터 경로 위에서 메시지를 중계**하는 미들웨어(인라인).
- DBTower = **데이터 경로 밖에서 관찰**하는 관제(아웃오브밴드).
- 성격이 정반대라 별도 저장소. 느슨한 연결(gwanmun 거래 로그를 DBTower가 관측)만 둔다.

### 전제 사실 (기술 배경)

| 전제 | 비고 |
|---|---|
| 국내 금융/공공은 여전히 고정길이 전문 + TCP 연계가 흔하다 | 대외계·코어뱅킹 고정길이 전문 관례 |
| 금융 전문의 한글은 EUC-KR/CP949 인코딩 | 문자 수가 아니라 byte 수로 다뤄야 하는 함정 |
| TCP는 스트림이라 한 전문이 여러 번에 나뉘어 온다 | partial read — 버퍼 누적·경계 관리 필요 |

## 공통 가드레일

1. **전문은 바이트가 진실** — 문자열로 대충 다루지 않는다. byte[] 오프셋·인코딩·패딩·정렬을 명시.
   한글 1자가 EUC-KR 2byte라 "길이 20"은 문자 수가 아니라 바이트 수(함정).
2. **실측** — 모든 Phase는 로컬 e2e(TCP 목업 계정계 포함) + **hex 덤프 스크린샷** + VERIFICATION 절.
3. **정직 범위** — 실제 금융 표준 전문 전수 구현 아님(대표 거래 몇 종). "표준 전수 아님"을 정직히 표기.
4. **읽고 판단 위임** — 관문은 흐름(인증·라우팅·변환)을 통제하되 **비즈니스 로직(잔액 계산 등)은
   계정계에 위임**한다. 새 REST 엔드포인트는 열지만 계산은 안 한다.
5. **커밋 한국어·이모지 금지·블로그 개선 아크.**

---

## Phase 1 — 전문(電文) 파서/빌더 (완료, 이 프로젝트의 심장)

**상황**: 계정계가 `"0200"+계좌번호[14]+거래코드[4]+...` 형태의 고정길이 바이트를 보낸다.
이걸 `{"account":"...","code":"..."}` JSON으로 바꿔야 앱이 알아듣는다. 반대도 마찬가지.

**한계 인지**:
- **인코딩**: 한글 필드는 EUC-KR 2byte. `String.substring(오프셋, 길이)`로 자르면 **한글이 깨진다** —
  문자 인덱스가 아니라 byte 인덱스로 잘라야 한다.
- **패딩·정렬**: 숫자는 우측 정렬·좌측 제로패딩, 문자는 좌측 정렬·우측 공백패딩이 관례. 혼동 시 파싱 붕괴.

**판단**: 필드 레이아웃을 **스펙으로 선언**(어노테이션: 이름·오프셋·길이·타입·패딩·인코딩)하고,
스펙을 읽어 byte[]↔DTO↔JSON을 양방향 변환하는 엔진을 만든다. 하드코딩 파싱 금지.

**개선**: 전문 스펙 정의(잔액조회 요청 30B·응답 61B) + Parser(byte[]→DTO) + Builder(DTO→byte[]) +
JSON 브릿지(Jackson).

**실측**: 대표 전문 2종 byte[]↔DTO↔JSON **왕복 무손실**(단위 테스트). 한글 EUC-KR 왕복.
잘린 전문·잘못된 길이 → 명확한 예외. hex 덤프 스크린샷.
**잔여**: 파싱만으론 연계 아님(네트워크 없음) — Phase 2.

---

## Phase 2 — 프레이밍 & 프로토콜 변환 (완료)

**상황**: 파서는 됐다. 이제 TCP로 들어온 전문을 받아 통역하고, 응답을 다시 전문으로 돌려줘야 한다.

**한계 인지**: **TCP는 스트림이라 한 전문이 한 번에 안 온다.** "고정길이 61바이트"인데 일부만 버퍼에
있으면 나머지를 기다려야 한다(partial read). 반대로 두 전문이 붙어 오기도 한다(뭉침).

**판단**: 프레이밍을 명확히 — 필요한 바이트가 다 모일 때까지 버퍼에 누적한 뒤에야 한 전문으로 넘긴다.
순수 `java.net` 소켓으로 직접 구현해 원리를 드러낸다.

**개선**: `FixedLengthFramer`(누적·완성 프레임 추출·뭉침 분리) + `FramedConnection` + 목업 계정계
TCP 서버 + 게이트웨이 왕복(REST→전문 build→TCP→응답 전문 parse→JSON).

**실측**: 두 JVM(앱↔목업계정계) 실제 소켓 왕복. partial read(전문을 나눠 전송) 재조립·뭉침·동시 N건.
요청/응답 전문 hex 덤프 스크린샷.
**잔여**: 외부에 그냥 열면 위험 — Phase 3.

---

## Phase 3 — API 게이트웨이 필터 체인 + 모듈러 모놀리스 (완료)

**상황**: 이 통로가 외부(핀테크·마이데이터)에 열린다. 아무나 계정계에 요청 못 하게, 과부하 못 주게
문지기가 필요하다.

**판단**: 필터 체인을 **직접 구현**(프레임워크 통짜 대신 손으로, 게이트웨이가 내부에서 뭘 하는지
이해하려고). 순서: 인증 → 라우팅 → 유량제어. 동시에 코드를 **Spring Modulith 모듈**로 경계화한다.

**개선**:
- 인증: API 키 검증 필터(미인증 401 / 잘못된 키 403).
- 라우팅: 경로/메서드 기반 라우트 테이블(모르는 경로 404).
- 유량제어: 클라이언트별 토큰버킷(초과 429 + Retry-After), 단조 시계·버킷 단위 동기화로 동시성 방어.
- 모듈 경계: message·core·gateway·web를 애플리케이션 모듈로 나누고 `ApplicationModules.verify()`로
  순환 없는 단방향 DAG를 코드가 강제하게 한다.

**실측**: 무키 401 / 잘못된 키 403 / 정상 통과 200 / 연속 전송 429 / 모르는 경로 404, 동시성 테스트.
`verify()` 그린 + 모듈 다이어그램.
**잔여**: JWT/OAuth·분산 rate limit 공유는 미구현(단일노드 인메모리) — 정직히 표기.

---

## Phase 4 — 가변길이 전문 + 커넥션 풀 (완료)

**상황**: 잔액조회는 고정 61byte였지만, 거래내역 조회 같은 응답은 **레코드 건수만큼 길이가 매번 다르다.**
그리고 요청마다 소켓을 새로 여는 건 낭비다.

**한계 인지**: 가변 전문은 본문 길이를 미리 모른다 — 길이 헤더를 먼저 읽어 본문 길이를 확정한 뒤
그만큼 더 모아야 한다(2단계 프레이밍). 커넥션 풀은 동시성·고갈 처리가 까다롭다.

**판단**: 길이 프리픽스 프레이밍(헤더 → 본문 2단계)과 커넥션 풀(최대 크기·유휴 반납·검증·고갈 거절)을
각각 core 모듈에 추가. 비정상 길이는 fail-closed로 방어.

**개선**: `LengthPrefixedFramer`/`LengthPrefixedConnection`, 가변 전문 스펙(고정 헤더 + 레코드 N건),
`ConnectionPool`(스레드 안전·슬롯 선점으로 초과 생성 차단) + 각 클라이언트 레트로핏.

**실측**: 헤더/본문 반쪽 도착 재조립·뭉침·비정상 길이 거절, 가변 레코드 N건 왕복, 풀 재사용(같은 소켓
재사용)·동시성·고갈 정책. `verify()` 계속 그린.
**잔여**: 커넥션 풀은 단일 대상 기준 — 다중 백엔드 풀링은 확장 지점.

---

## Phase 5 — 잔여 (정직하게 안 하는 것)

- **금융 표준 전문 전수**: 은행별 실제 규격 전부는 범위 밖. 대표 거래 몇 종만.
- **트랜잭션 모니터링·DBTower 연동**: 거래 로그 적재·실패 분류·관제 연동은 확장 지점(개념만).
- **HA·이중화**: 단일 인스턴스. 부하분산·페일오버는 요구 생기면.
- **MQ 연동**: TCP/HTTP만 구현. Kafka/MQ 어댑터는 확장 지점.
- **전문 암호화·전용선**: 실제 금융망 보안(VPN·전문 암호화)은 인프라 영역 — 개념만 언급.
