# gwanmun(관문) 로드맵 — 상황 주도 개선 아크

> 기능 나열이 아니라 **"어떤 상황에서 무엇이 깨지고, 그래서 무엇을 만드는가"**로 쓴다.
> 각 Phase = (상황) → (한계) → (판단) → (개선) → (라이브 실측·hex 덤프) → (정직한 잔여).

---

## 0. 이 프로젝트가 존재하는 이유 — 한 상황에서 출발

**상황**: 은행에 모바일 잔액조회를 붙여야 한다. 계정계(코어뱅킹)는 **고정길이 전문(電文) + TCP**로만
말하고, 모바일 앱은 **JSON + HTTP REST**로만 말한다. 둘은 직접 대화가 안 되는데, 계정계를 통째로
뜯어고치는 건 현실적으로 감당하기 어려운 규모다.

**판단**: 두 시스템을 못 고치니 **중간에 통역기를 세운다.** 두 층으로 된 미들웨어 —
①**API 게이트웨이**(외부에 열린 통로의 문지기: 인증·라우팅·유량제어) + ②**연계 통역**(전문↔JSON,
TCP↔HTTP 변환). 마이데이터·오픈뱅킹이 정확히 이 구조다(외부 핀테크가 REST로 요청 → 계정계는 전문).

**"관문" = 옛 시스템과 새 앱 사이의 동시통역사 + 검문소.**

### 정체성 경계 (DBTower와 안 섞음)

- gwanmun = **데이터 경로 위에서 메시지를 중계**하는 미들웨어(인라인).
- DBTower = **데이터 경로 밖에서 관찰**하는 관제(아웃오브밴드).
- 성격이 정반대라 별도 저장소. 느슨한 연결(gwanmun 거래 로그를 DBTower가 관측)만 둔다.

### 전제 사실 (기술 배경)

| 전제 | 비고 |
|---|---|
| 국내 금융/공공은 여전히 고정길이 전문 + TCP 연계가 흔하다 | 대외계·코어뱅킹 고정길이 전문 관례 |
| 금융 전문의 한글은 EUC-KR/CP949 인코딩 | 문자 수가 아니라 byte 수로 다뤄야 하는 함정 |
| TCP는 스트림이라 한 전문이 여러 번에 나뉘어 온다 | partial read — 버퍼 누적·경계 관리 필요 |

## 공통 가드레일

1. **전문은 바이트가 진실** — 문자열로 대충 다루지 않는다. byte[] 오프셋·인코딩·패딩·정렬을 명시.
   한글 1자가 EUC-KR 2byte라 "길이 20"은 문자 수가 아니라 바이트 수(함정).
2. **실측** — 모든 Phase는 로컬 e2e(TCP 목업 계정계 포함) + **hex 덤프 스크린샷** + VERIFICATION 절.
3. **정직 범위** — 실제 금융 표준 전문 전수 구현 아님(대표 거래 몇 종). "표준 전수 아님"을 정직히 표기.
4. **읽고 판단 위임** — 관문은 흐름(인증·라우팅·변환)을 통제하되 **비즈니스 로직(잔액 계산 등)은
   계정계에 위임**한다. 새 REST 엔드포인트는 열지만 계산은 안 한다.
5. **커밋 한국어·이모지 금지·블로그 개선 아크.**

## Phase 1 — 전문(電文) 파서/빌더 (완료, 이 프로젝트의 심장)

**상황**: 계정계가 `"0200"+계좌번호[14]+거래코드[4]+...` 형태의 고정길이 바이트를 보낸다.
이걸 `{"account":"...","code":"..."}` JSON으로 바꿔야 앱이 알아듣는다. 반대도 마찬가지.

**한계 인지**:
- **인코딩**: 한글 필드는 EUC-KR 2byte. `String.substring(오프셋, 길이)`로 자르면 **한글이 깨진다** —
  문자 인덱스가 아니라 byte 인덱스로 잘라야 한다.
- **패딩·정렬**: 숫자는 우측 정렬·좌측 제로패딩, 문자는 좌측 정렬·우측 공백패딩이 관례. 혼동 시 파싱 붕괴.

**판단**: 필드 레이아웃을 **스펙으로 선언**(어노테이션: 이름·오프셋·길이·타입·패딩·인코딩)하고,
스펙을 읽어 byte[]↔DTO↔JSON을 양방향 변환하는 엔진을 만든다. 하드코딩 파싱 금지.

**개선**: 전문 스펙 정의(잔액조회 요청 30B·응답 61B) + Parser(byte[]→DTO) + Builder(DTO→byte[]) +
JSON 브릿지(Jackson).

**실측**: 대표 전문 2종 byte[]↔DTO↔JSON **왕복 무손실**(단위 테스트). 한글 EUC-KR 왕복.
잘린 전문·잘못된 길이 → 명확한 예외. hex 덤프 스크린샷.
**잔여**: 파싱만으론 연계 아님(네트워크 없음) — Phase 2.

## Phase 2 — 프레이밍 & 프로토콜 변환 (완료)

**상황**: 파서는 됐다. 이제 TCP로 들어온 전문을 받아 통역하고, 응답을 다시 전문으로 돌려줘야 한다.

**한계 인지**: **TCP는 스트림이라 한 전문이 한 번에 안 온다.** "고정길이 61바이트"인데 일부만 버퍼에
있으면 나머지를 기다려야 한다(partial read). 반대로 두 전문이 붙어 오기도 한다(뭉침).

**판단**: 프레이밍을 명확히 — 필요한 바이트가 다 모일 때까지 버퍼에 누적한 뒤에야 한 전문으로 넘긴다.
순수 `java.net` 소켓으로 직접 구현해 원리를 드러낸다.

**개선**: `FixedLengthFramer`(누적·완성 프레임 추출·뭉침 분리) + `FramedConnection` + 목업 계정계
TCP 서버 + 게이트웨이 왕복(REST→전문 build→TCP→응답 전문 parse→JSON).

**실측**: 두 JVM(앱↔목업계정계) 실제 소켓 왕복. partial read(전문을 나눠 전송) 재조립·뭉침·동시 N건.
요청/응답 전문 hex 덤프 스크린샷.
**잔여**: 외부에 그냥 열면 위험 — Phase 3.

## Phase 3 — API 게이트웨이 필터 체인 + 모듈러 모놀리스 (완료)

**상황**: 이 통로가 외부(핀테크·마이데이터)에 열린다. 아무나 계정계에 요청 못 하게, 과부하 못 주게
문지기가 필요하다.

**판단**: 필터 체인을 **직접 구현**(프레임워크 통짜 대신 손으로, 게이트웨이가 내부에서 뭘 하는지
이해하려고). 순서: 인증 → 라우팅 → 유량제어. 동시에 코드를 **Spring Modulith 모듈**로 경계화한다.

**개선**:
- 인증: API 키 검증 필터(미인증 401 / 잘못된 키 403).
- 라우팅: 경로/메서드 기반 라우트 테이블(모르는 경로 404).
- 유량제어: 클라이언트별 토큰버킷(초과 429 + Retry-After), 단조 시계·버킷 단위 동기화로 동시성 방어.
- 모듈 경계: message·core·gateway·web를 애플리케이션 모듈로 나누고 `ApplicationModules.verify()`로
  순환 없는 단방향 DAG를 코드가 강제하게 한다.

**실측**: 무키 401 / 잘못된 키 403 / 정상 통과 200 / 연속 전송 429 / 모르는 경로 404, 동시성 테스트.
`verify()` 그린 + 모듈 다이어그램.
**잔여**: JWT/OAuth·분산 rate limit 공유는 미구현(단일노드 인메모리) — 정직히 표기.

## Phase 4 — 가변길이 전문 + 커넥션 풀 (완료)

**상황**: 잔액조회는 고정 61byte였지만, 거래내역 조회 같은 응답은 **레코드 건수만큼 길이가 매번 다르다.**
그리고 요청마다 소켓을 새로 여는 건 낭비다.

**한계 인지**: 가변 전문은 본문 길이를 미리 모른다 — 길이 헤더를 먼저 읽어 본문 길이를 확정한 뒤
그만큼 더 모아야 한다(2단계 프레이밍). 커넥션 풀은 동시성·고갈 처리가 까다롭다.

**판단**: 길이 프리픽스 프레이밍(헤더 → 본문 2단계)과 커넥션 풀(최대 크기·유휴 반납·검증·고갈 거절)을
각각 core 모듈에 추가. 비정상 길이는 fail-closed로 방어.

**개선**: `LengthPrefixedFramer`/`LengthPrefixedConnection`, 가변 전문 스펙(고정 헤더 + 레코드 N건),
`ConnectionPool`(스레드 안전·슬롯 선점으로 초과 생성 차단) + 각 클라이언트 레트로핏.

**실측**: 헤더/본문 반쪽 도착 재조립·뭉침·비정상 길이 거절, 가변 레코드 N건 왕복, 풀 재사용(같은 소켓
재사용)·동시성·고갈 정책. `verify()` 계속 그린.
**잔여**: 커넥션 풀은 단일 대상 기준 — 다중 백엔드 풀링은 확장 지점.

## Phase 5 — 관측 가능한 거래 원장 (완료)

**상황**: 통로는 튼튼해졌는데, 게이트웨이가 거래를 흘려보내기만 하고 **아무것도 기억하지 못한다.**
"어제 그 이체 요청이 어떻게 됐나요?"에 답할 방법이 없다 — 앱 로그를 뒤져도 어떤 로그가 어떤 요청인지,
그 요청이 계정계까지 갔는지, 응답을 받았는지 알 수 없다. 장애가 나면 대응 불가다.

**한계 인지**:
- **타임아웃 = 실패가 아니다.** 응답을 못 받은 거래를 임의로 FAILED로 적으면, 실패인 줄 알고 재시도해
  이중 거래가 난다. 계정계에서는 처리됐을 수 있다 — 상태는 2값이 아니라 **3값**이어야 한다.
- **적재가 거래를 막으면 안 된다.** 원장 DB가 죽었다고 거래까지 죽으면 관측이 장애 원인이 된다.
- **원장에 계좌 원문을 쌓으면 안 된다.** 거래 기록이 곧 민감정보 창고가 된다.

**판단**: 새 모듈 `ledger`를 세운다 —
- **거래고유번호 채번**: 오픈뱅킹 bank_tran_id의 자기설명 구조 참조(생성주체 GWMN + 구분 U + 날짜8 +
  일련번호9). AtomicLong(스레드 안전) + 자정 이후 ms 기반 시드(재기동 안전).
- **3값 상태**: 응답 정상 수신=SUCCESS / 명확한 오류 응답·입력 오류·연결 거부=FAILED /
  **타임아웃·응답 없는 끊김=UNKNOWN**. 원인 예외 사슬로 판정.
- **비동기 적재**: 전용 스레드 + 유한 큐. 큐 포화·DB 장애에도 거래는 진행(WARN만).
- **마스킹**: 저장 직전 한 곳에서 앞6+뒤4만 노출. 앱 로그의 계좌도 같은 규칙.
- **correlation ID**: 수신 헤더 승계(없으면 생성) → MDC로 전 로그 라인 + 응답 헤더 + 원장.
- **관측**: actuator 헬스(liveness/readiness 분리) + Prometheus 커스텀 메트릭(토큰버킷 소비/거절,
  풀 active/idle/opened/reused, TCP 왕복 타이머, 거래 상태별 카운터) + graceful shutdown.
- 저장소는 H2(로컬·테스트)/PostgreSQL 컨테이너(운영 데모, docker-compose 포트 25432).

**실측**: 정상→SUCCESS / 없는 계좌(0001)→FAILED / 지연 모드 계좌(목업이 응답을 5초 지연, read
타임아웃 3초)→**UNKNOWN**이 원장에 실제 기록. PostgreSQL 행 직접 확인(계좌는 마스킹 형태만).
/actuator/prometheus 커스텀 메트릭 노출. 화면 스크린샷(3값 색상). `verify()` 계속 그린(5모듈 DAG).

**잔여**: 망취소(취소 전문)·거래 상태 조회(대사) 미구현 — UNKNOWN을 적는 것까지가 이번 범위이고,
UNKNOWN을 **해소**하는 건 다음 단계. 멱등키 없음(같은 요청 재시도 구분 불가). 채번은 단일 노드 전제.

## Phase 6 — 장애 내성 + UNKNOWN 해소 (완료)

**상황**: 두 개의 폭탄이 남아 있었다. 하나, 계정계가 죽거나 느려지면 **모든 호출이 타임아웃까지
스레드를 붙잡아** 게이트웨이가 같이 눕는다(장애 전파). 둘, Phase 5의 원장은 UNKNOWN을 **기록만
하고 방치한다** — "모른다"고 적어 두는 것까지가 전부였고, 모르는 거래를 확정 짓는 절차가 없었다.

**한계 인지**:
- **재시도는 양날이다.** 응답을 못 받은 거래를 무조건 재전송하면, 계정계에서 이미 처리된 변경성
  거래가 두 번 실행된다(이중 거래). 재시도 허용 여부는 옵션이 아니라 거래의 성격(조회성/변경성)에
  못 박아야 한다.
- **소켓 read 타임아웃은 "호출 한 번"의 제한일 뿐이다.** 재시도가 붙으면 거래 하나가 read 타임아웃 ×
  시도 횟수 + 백오프로 늘어진다 — 거래 전체의 데드라인이 따로 필요하고, 마지막 시도의 read 제한도
  남은 시간으로 깎여야 한다.
- **서킷 HALF_OPEN은 동시성 함정이다.** 회복 확인 전에 트래픽을 다 열면 살아나던 백엔드를 다시
  눕힌다 — 탐침 수를 제한하고, 탐침이 나가 있는 동안의 호출은 즉시 거절해야 한다.
- **원거래 전문에 거래고유번호가 없으면 대사가 불가능하다.** 게이트웨이 원장만 아는 ID로는 계정계에
  "그 거래 처리했습니까"를 물을 수 없다 — 열쇠가 전문 본문에 실려야 한다.

**판단**:
- 서킷브레이커를 **직접 구현**(프레임워크 통짜 대신 손으로 — Resilience4j 안 씀). CLOSED→(연속 실패
  임계)→OPEN(즉시 실패 — 원장 FAILED, 계정계 호출 없음)→(대기)→HALF_OPEN(제한 탐침)→CLOSED.
  동시성은 synchronized 한 겹, 상태 전이는 로그 + Prometheus 게이지/카운터로 노출.
- 재시도는 `TransactionKind`(INQUIRY/MUTATION)로 성격을 타입에 박고, **조회성만** 지수 백오프 재시도.
  변경성은 코드가 0회를 강제 — 응답을 못 받으면 UNKNOWN으로 남겨 해소 절차로 넘긴다.
- 거래 단위 데드라인을 소켓 read 타임아웃과 별개로 둔다(기본 5초). 새 시도의 출발선을 막을 뿐
  아니라 매 시도의 read 타임아웃을 남은 시간으로 깎는다.
- UNKNOWN 해소는 금융 연계의 정수를 따른다 — **응답을 못 받은 거래는 "모름"이고, 확정은 조회로
  한다.** 요청 전문 공통 선두를 `전문구분(4)+거래고유번호(22)`로 확장(30→52byte)하고, 목업 계정계에
  인메모리 원장(처리 거래 기억)과 전문 2종을 추가: 거래상태조회(0400 — 처리됨/미처리)와
  망취소(0420 — 원거래 취소, 멱등). 상태조회가 "처리됨"이면 망취소로 무효화해 CANCELED(호출자는
  이미 오류를 받았으므로 성공을 살려두면 양쪽 장부가 어긋난다 — 오픈뱅킹 망취소 관례), "미처리"면
  그제야 FAILED로 확정. 원장에 해소 시각·방법을 남긴다.

**개선**: `CircuitBreaker`/`ResilientExecutor`/`TransactionKind`(core), 해소 전문 4종(message),
`TransactionResolutionService`(gateway), `TransactionStatus.CANCELED`+해소 이력(ledger),
`POST /api/gateway/resolve/{tranId}`+서킷 상태 API+화면 섹션(web). 서킷 열림 응답은 503으로 구분.

**실측**: 계정계 프로세스를 실제로 죽여 연속 호출 → 502(임계 도달, 한 요청의 재시도 3회가 서킷을
엶) → 503×4(즉시 거절, 원장 elapsed_ms=0 FAILED) → 재기동 → HALF_OPEN 탐침 → CLOSED 회복(전이
로그·메트릭). 지연 계좌 UNKNOWN(5.02초 — 데드라인이 마지막 재시도를 깎음) → 상태조회 "처리됨" →
망취소 → 원장 CANCELED. 유실 계좌 UNKNOWN → "미처리" → FAILED. psql로 해소 이력까지 확인.
`./gradlew test` 129건 그린, `verify()` 그린. [VERIFICATION Phase 6절](VERIFICATION.md).
실측 중 함정 둘도 정직 기록: `ddl-auto:update`가 체크 제약을 갱신 안 해 CANCELED UPDATE가 거부된
것(수동 ALTER), 그 실패 순간 이미 나간 망취소를 멱등성이 구해 준 것.

**잔여**: 해소는 수동 API만(주기 대사 배치는 확장 지점). 멱등키 없음. 응답 전문에 거래ID 없음
(동기 소켓 전제). 서킷은 인스턴스 로컬.

## Phase 7 — 잔여 (정직하게 안 하는 것)

- **금융 표준 전문 전수**: 은행별 실제 규격 전부는 범위 밖. 대표 거래 몇 종만.
- **자동 대사 배치**: UNKNOWN 해소는 수동 트리거까지. 주기 스케줄러·대사 리포트는 다음 단계.
- **멱등키**: 호출자 재시도를 게이트웨이가 구분하는 흐름(Idempotency-Key)은 미구현.
- **DBTower 연동**: 거래 원장을 관제가 관측하는 느슨한 연결은 확장 지점(개념만).
- **HA·이중화**: 단일 인스턴스. 부하분산·페일오버·중앙 채번·서킷 상태 공유는 요구 생기면.
- **MQ 연동**: TCP/HTTP만 구현. Kafka/MQ 어댑터는 확장 지점.
- **전문 암호화·전용선**: 실제 금융망 보안(VPN·전문 암호화)은 인프라 영역 — 개념만 언급.
- **스키마 마이그레이션 도구**: `ddl-auto:update`의 한계(체크 제약 미갱신)를 실측했다 — Flyway류 도입은 확장 지점.
