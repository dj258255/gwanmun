# gwanmun(관문) 로드맵 — 상황 주도 개선 아크

> 기능 나열이 아니라 **"어떤 상황에서 무엇이 깨지고, 그래서 무엇을 만드는가"**로 쓴다.
> 각 Phase = (상황 가정) → (한계) → (판단) → (개선) → (라이브 실측·hex 덤프 스크린샷) → (정직한 잔여).
> 인젠트 연계 직무 담당업무를 층별로 정면 대응. 웹서칭으로 실전 함정 검증 후 적었다.

---

## 0. 이 프로젝트가 존재하는 이유 — 한 상황에서 출발

**상황 가정**: 은행에 모바일 잔액조회를 붙여야 한다. 계정계(코어뱅킹)는 20년 된 시스템이라
**고정길이 전문(電文) + TCP**로만 말한다. 모바일 앱은 **JSON + HTTP REST**로만 말한다.
둘은 직접 대화가 안 된다. 계정계를 뜯어고치는 건 수백억짜리 프로젝트라 불가능하다.

**한계 인지**: 두 시스템을 못 고친다. 그러면 **중간에 통역기를 세우는 수밖에 없다.**

**판단**: 두 층으로 된 미들웨어를 만든다 — ①**API 게이트웨이**(외부에 열린 통로의 문지기:
인증·라우팅·유량제어) + ②**연계 통역**(전문↔JSON, TCP↔HTTP 변환). 마이데이터·오픈뱅킹이
정확히 이 상황이다(외부 핀테크가 REST로 요청 → 계정계는 전문). 인젠트가 iGate(연계)+APIM(API GW)을
함께 파는 이유이자, 이 프로젝트가 둘을 한 흐름에 담는 이유.

**"관문" = 옛 시스템과 새 앱 사이의 동시통역사 + 검문소.**

### 정체성 경계 (중요 — DBTower와 안 섞음)

- gwanmun = **데이터 경로 위에서 메시지를 중계**하는 미들웨어(인라인).
- DBTower = **데이터 경로 밖에서 관찰**하는 관제(아웃오브밴드).
- 성격이 정반대라 별도 저장소. 느슨한 연결(gwanmun 거래 로그를 DBTower가 관측)만 둔다.

### 전제 사실 (2026-07-08 웹 검증 — 지어낸 것 없음)

| 전제 | 근거 | 상태 |
|---|---|---|
| 국내 금융/공공은 여전히 고정길이 전문+TCP | 대외계(FEP)·코어뱅킹 고정길이 전문 표준, 전용선/VPN | 웹 확인 |
| 한글은 EUC-KR/CP949 인코딩(전문 함정) | 금융 전문 한글 EUC-KR 관례 | 웹 확인 |
| 인젠트는 iGate(연계)+APIM(API GW) 둘 다 판다 | 회사 조사(GS인증 1등급 APIM 등) | 웹 확인 |
| TCP는 스트림이라 한 전문이 여러 번에 나뉘어 온다 | Netty 프레임 디코더 문서(partial read·버퍼 누적) | 웹 확인 |

## 공통 가드레일

1. **전문은 바이트가 진실** — 문자열로 대충 다루지 않는다. byte[] 오프셋·인코딩·패딩·정렬을 명시.
   한글 1자가 EUC-KR 2byte라 "길이 20"은 문자 수가 아니라 바이트 수(함정).
2. **실측** — 모든 Phase는 로컬 e2e(TCP 목업 계정계 포함) + **hex 덤프 스크린샷** + VERIFICATION 절.
3. **정직 범위** — 실제 금융 표준 전문 전수 구현 아님(대표 거래 몇 종). "학습판"이지 상용 iGate와 동급 주장 금지.
4. **읽고 판단 위임 원칙** — 관문은 흐름(인증·라우팅·변환)을 통제하되 **비즈니스 로직(잔액 계산 등)은
   계정계에 위임**한다. 새 REST 엔드포인트는 열지만 계산은 안 한다.
5. **커밋 한국어·이모지 금지·블로그 개선 아크.**

## Phase 1 — 전문(電文) 파서/빌더 (이 프로젝트의 심장)

**상황**: 계정계가 `"0200"+계좌번호[14]+거래코드[4]+...` 형태의 고정길이 바이트를 보낸다.
이걸 `{"account":"...","code":"..."}` JSON으로 바꿔야 앱이 알아듣는다. 반대도 마찬가지.

**한계 인지 (웹 검증된 실제 함정)**:
- **인코딩**: 한글 필드는 EUC-KR 2byte. `String.substring(오프셋, 길이)`로 자르면 **한글이 깨진다** —
  문자 인덱스가 아니라 byte 인덱스로 잘라야 한다.
- **패딩·정렬**: 숫자는 우측 정렬·좌측 제로패딩, 문자는 좌측 정렬·우측 공백패딩이 관례. 혼동 시 파싱 붕괴.
- **부호**: 금액 음수(마지막 자리 오버펀치 등 레거시 관례)는 복잡 — 범위 밖으로 두되 문서화.

**판단**: 필드 레이아웃을 **스펙으로 선언**(어노테이션 또는 YAML: 이름·오프셋·길이·타입·패딩·인코딩)하고,
스펙을 읽어 byte[]↔DTO↔JSON을 양방향 변환하는 엔진을 만든다. 하드코딩 파싱 금지(전문마다 새로 짜면 지옥).

**개선(구현)**:
- 전문 스펙 정의 예: 잔액조회 요청 = 전문구분[4]+계좌번호[14]+거래코드[4]+필러[N],
  응답 = ...+잔액[15]+응답코드[4].
- Parser: byte[] → DTO(오프셋 슬라이싱, EUC-KR 디코딩, 패딩 제거).
- Builder: DTO → byte[](길이 고정·패딩·인코딩 역변환).
- JSON 브릿지: DTO ↔ JSON(Jackson).

**실측(라이브)**: 대표 전문 2~3종에 대해 byte[]↔DTO↔JSON **왕복 무손실**(round-trip 단위 테스트).
한글 포함 필드 EUC-KR 왕복. 잘린 전문·잘못된 길이 → 명확한 예외. **hex 덤프로 전문 스크린샷.**
**잔여**: 파싱만으론 연계 아님(네트워크 없음) — Phase 2.
**산출물**: VERIFICATION 1절 · 블로그 1편 "바이트가 진실이다 — 고정길이 전문 파서".

## Phase 2 — 어댑터 & 프로토콜 변환

**상황**: 파서는 됐다. 이제 TCP로 들어온 전문을 받아 → JSON으로 → HTTP로 백엔드에 넘기고 →
응답을 다시 전문으로 → TCP로 돌려줘야 통역이 완성된다.

**한계 인지 (웹 검증)**: **TCP는 스트림이라 한 전문이 한 번에 안 온다.** 절반만 왔다가 나중에 나머지가
온다(partial read). "고정길이 200바이트"인데 120바이트만 버퍼에 있으면 기다려야 한다 —
[Netty 프레임 디코더](https://netty.io/4.1/api/io/netty/handler/codec/LengthFieldBasedFrameDecoder.html)가
이 문제를 위해 존재. 직접 짜면 버퍼 누적·경계 관리를 손으로 해야 함.

**판단**: 프레이밍을 명확히 — 고정길이면 길이 상수, 가변이면 길이 헤더. Netty의
FixedLength/LengthFieldBasedFrameDecoder를 쓰되, **왜 필요한지(partial read)를 이해하고** 쓴다
(면접에서 "TCP 스트림 왜 프레이밍 하냐" 답할 수 있게).

**개선(구현)**:
- inbound TCP 서버(Netty): 프레이밍으로 전문 완성 → Parser → DTO → JSON.
- outbound: JSON을 내부 REST 백엔드(모의 코어뱅킹)로 WebClient 호출 → 응답 JSON → Builder로 전문 → TCP 응답.
- 레거시 목업: 전문 요청→전문 응답 주는 작은 TCP 서버(진짜 은행 없이 양방향 e2e).
- 거래코드 → 라우팅 룰(코드별 대상·매핑) 설정화.

**실측(라이브)**: e2e — TCP로 잔액조회 전문 전송 → gwanmun이 JSON 변환·REST 호출·전문 응답 반환.
**요청/응답 전문 hex 덤프 스크린샷.** partial read 유발(전문을 2번에 나눠 전송)해도 정상 처리 확인. 동시 N건.
**잔여**: 외부에 그냥 열면 위험 — Phase 3.
**산출물**: VERIFICATION 2절 · 블로그 2편 "TCP 전문을 REST로, 다시 전문으로".

## Phase 3 — API 게이트웨이 층 (원래 만들고 싶던 것)

**상황**: 이 통로가 외부(핀테크·마이데이터)에 열린다. 아무나 계정계에 요청 못 하게, 과부하 못 주게
문지기가 필요하다. (API 게이트웨이 = 아파트 경비실: 신분확인·안내·방문횟수 제한, 집 안 살림엔 관여 안 함.)

**한계 인지**: 통역만 하고 인증·유량제어가 없으면, 외부에 여는 순간 무방비다. 각 백엔드가 따로 인증하면
중복이고 관리 안 됨 → 관문 앞단에서 한 번에.

**판단**: Spring 필터 체인으로 **직접 구현**(Spring Cloud Gateway 통짜 사용은 지양 — 게이트웨이가
내부에서 뭘 하는지 손으로 이해하는 게 학습 목적). 필터 순서: 인증 → 라우팅 → 유량제어 → 로깅.

**개선(구현)**:
- 인증: JWT 검증 필터(만료·서명). 미인증 401.
- 라우팅: 경로/헤더 기반으로 내부 서비스(또는 Phase 2 연계층) 라우팅.
- 유량제어: 토큰버킷 레이트리밋(클라이언트별 분당 N). 초과 429.
  (DBTower WebhookNotifier 레이트리밋에서 밟은 스레드 안전 패턴 재사용.)
- 로깅/추적: correlationId 부여, 요청/응답 로깅(계좌·잔액 등 민감정보 마스킹).
- (선택) 서킷브레이커: 백엔드 장애 시 빠른 실패.

**실측(라이브)**: 무토큰 401 / 만료토큰 401 / 정상 통과 / 분당 초과 429 / 백엔드 다운 시 빠른 실패.
각각 스크린샷·로그.
**잔여**: 거래가 어디서 깨졌나 안 보임 — Phase 4.
**산출물**: VERIFICATION 3절 · 블로그 3편 "게이트웨이를 손으로 — 인증·라우팅·유량제어".

## Phase 4 — 트랜잭션 모니터링·장애 대응

**상황**: 운영 중 "잔액조회가 가끔 실패한다"는 문의가 왔다. 어디서 깨졌나(파싱? 백엔드 타임아웃?
인증?) 안 보이면 대응을 못 한다. JD의 "트랜잭션 모니터링·장애 대응"이 이것.

**한계 인지**: 거래가 5개 층(인증→라우팅→변환→백엔드→역변환)을 통과하는데, 실패 지점이 안 보이면
장애 대응이 추측이 된다.

**개선(구현)**:
- 거래 로그(MySQL): 거래ID·거래코드·인바운드/아웃바운드 시각·상태·에러·소요ms.
- 실패 분류: 파싱 실패 / 백엔드 타임아웃 / 인증 실패 / 매핑 오류 — 종류별 카운트.
- 조회 API 또는 간단 대시보드: 최근 거래·실패율·거래코드별 지연.
- **함정**: 로깅이 처리 지연을 유발하지 않게(비동기). 민감 필드 마스킹.
- **(다리) DBTower 연동**: gwanmun의 MySQL 거래 로그를 DBTower에 인스턴스로 등록 → 관제탑이 이
  게이트웨이의 트랜잭션까지 관측(두 프로젝트가 실무 구조로 연결, 도그푸딩).

**실측(라이브)**: 정상·타임아웃·파싱오류 거래를 각각 유발 → 대시보드에 분류·집계 → 스크린샷.
DBTower 연동 시 관제탑에서 gwanmun 거래 쿼리가 보이는 스크린샷.
**산출물**: VERIFICATION 4절 · 블로그 4편 "거래가 어디서 깨졌나 — 연계 모니터링".

## Phase 5 — 잔여 (정직하게 안 하는 것)

- **금융 표준 전문 전수**: 금융결제원·은행별 실제 규격 전부는 범위 밖. 대표 거래 몇 종만 — 명시.
- **HA·이중화**: 단일 인스턴스. 부하분산·페일오버는 요구 생기면(DBTower ShedLock 경험은 별도 자산, 섞지 않음).
- **MQ 연동(EAI 허브)**: TCP/HTTP만 구현. Kafka/MQ 어댑터는 확장 지점.
- **전문 암호화·전용선**: 실제 금융망 보안(VPN·전문 암호화)은 인프라 영역 — 개념만 언급.

## 우선순위 (마감 7/12 고려)

- **서류(지금)**: 관문 미완성이어도 지원 가능. 자소서에 "고정길이 전문 파서 구현 중"(로드맵 실존 → 거짓 아님).
- **면접 전(서류 합격 후)**: **Phase 1(전문 파서) + Phase 2(TCP↔HTTP) 최소 e2e**를 우선 완성 —
  이 둘이 인젠트 유일 갭(Flat Data)을 정면으로 메운다. Phase 3~4는 그다음.
